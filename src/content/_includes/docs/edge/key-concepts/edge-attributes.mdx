Edge stores attributes locally in its PostgreSQL database, making them available for rule logic, dashboards, and external integrations even when the server connection is unavailable.

ThingsBoard supports three attribute scopes:

- **Server attributes:** Set by the server or a user. Not reported by the device. Used for configuration, metadata, and context — for example, an installation location or a threshold value.
- **Client attributes:** Reported by the device itself, like firmware version or battery level. Read-only from the platform side.
- **Shared attributes:** Set by the server and pushed to the device when it subscribes. Used to send configuration or commands to devices.

## Attribute synchronization

By default, attribute changes are not automatically propagated between Edge and the server. Synchronization requires configuring the rule chain on each side to connect the **Attributes Updated** and **Attributes Deleted** output relations to the corresponding **push to cloud** or **push to edge** node. For details, see the Edge attributes guide.

## Edge instance attributes

Attributes can be assigned to the **Edge instance itself** — not just to devices or assets. This is a distinct concept introduced in release 3.9. Edge instance attributes act as placeholders in rule chains, letting a single rule chain template serve multiple Edge instances with different configurations at runtime — eliminating the need for a separate template per instance.

For example, you can store an `edgeAccessToken` attribute on the Edge instance and reference it in a REST API call node via metadata — so the same rule chain works across Edge deployments with different tokens.
