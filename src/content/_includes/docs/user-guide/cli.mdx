**ThingsBoard CLI** is a command-line tool (`thingsboard`, alias `tb`) for managing your ThingsBoard resources from the terminal.

It is purpose-built for **AI-driven solution development**: a single `tb init` call scaffolds a project on disk together with [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview) skills and agents that teach Claude how to design, generate, and deploy complete IoT solutions for you. You can also drive the CLI directly from the terminal for scripting, CI, or one-off tasks.

## Key concepts

- **Project** — your local workspace for designing and managing ThingsBoard solutions. You create one with `tb init` in any folder you like. From then on, that folder is where Claude builds your solutions and where the CLI looks when you run commands.
- **Solution** — a self-contained collection of devices, dashboards, alarms, and the rest that work together to solve one business need (a smart house, a fleet, a factory line). A single project can hold many solutions, and the same solution can be deployed to multiple ThingsBoard targets (dev, staging, prod) without any changes.

## Two ways to use it

1. **Let Claude drive it.** Open the project in Claude Code, describe the solution you want, review what Claude produced, then run `tb push` to deploy.
2. **Drive it yourself.** Run `tb device save`, `tb dashboard save`, `tb push`, and friends directly — great for scripting, CI pipelines, or quick one-off tasks.

## Prerequisites

- A ThingsBoard account — ThingsBoard Cloud, ThingsBoard Cloud EU, or self-hosted PE.
- An <DocLink product={props.product} path="user-guide/security/api-keys" useTbDocs>API key</DocLink> with CRUD permissions for the resources the CLI will manage.

## Installation

<Steps>
1. **Install uv.** [uv](https://docs.astral.sh/uv/) is a fast Python package manager that handles the CLI's virtual environment for you.

   ```bash
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```

   See the [uv installation guide](https://docs.astral.sh/uv/getting-started/installation/) for platform-specific options.

2. **Install the CLI.**

   ```bash
   uv tool install thingsboard-cli
   ```

   This installs the `thingsboard` command and its `tb` alias on your system.

3. **Verify the install.**

   ```bash
   tb --version
   ```

4. **Keep it up to date.** The CLI checks for updates automatically each time you run it. You can also upgrade manually with `tb update`.
</Steps>

## Configure your first profile

Run the interactive wizard:

```bash
tb config
```

The wizard asks four questions, in this order:

```text
ThingsBoard URL [https://thingsboard.cloud]:
Auth method [api_key]:
API Key: ********
Profile name: cloud
```

- **ThingsBoard URL.** Enter the URL of the ThingsBoard instance you want to connect to:
  - `https://thingsboard.cloud` — ThingsBoard Cloud (North America). This is the default; just press Enter.
  - `https://eu.thingsboard.cloud` — ThingsBoard Cloud EU.
  - Your own URL (e.g. `https://tb.example.com`) — self-hosted ThingsBoard PE.
- **Auth method.** `api_key` is recommended. The alternative `jwt` prompts for username and password, which are stored in plaintext — the CLI will warn you and suggest API key auth instead.
- **Profile name.** The wizard saves the profile under this name and makes it your default. Pick anything that makes sense for you — `cloud`, `cloud-eu`, `prod`, etc.

On success the wizard prints:

```text
Profile '<name>' saved. Connected to <url>
```

If the connectivity test fails, the CLI still saves the profile but prints a warning. Re-run `tb config` to fix the URL or key, or run `tb config show` to confirm what's active.

For scripting and CI, skip the wizard and use the one-shot form:

```bash
tb config set --url <url> --api-key <key>
```

You can have any number of profiles (typically one per ThingsBoard target):

```bash
tb config list                                       # show all profiles
tb config use prod                                   # switch the default profile
tb config add staging --url https://... --api-key ...
tb config remove staging
```

Pass `--profile <name>` to any command to override the active profile for that single call.

## Create your first project

```bash
mkdir my-iot-project && cd my-iot-project && tb init
```

The scaffold creates a directory tree like this:

```text
my-iot-project/
  thingsboard.json         # project config
  .gitignore
  .git/                    # git repository (initialized if not already inside one)
  devices/                 # entity directories — one JSON file per entity
  device-profiles/
  dashboards/
  rule-chains/
  customers/
  calculated-fields/
  ...
  solutions/               # solutions live here, one subdirectory per solution
  tasks/                   # synthetic-telemetry generators
  .claude/                 # Claude Code skills bundled with the CLI
```

Open the project in [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview):

```bash
claude --model sonnet
```

The scaffolded `.claude/` skills auto-load and Claude can immediately design and generate solutions for you.

## Building solutions with Claude

Once you have a project on disk, the AI-first workflow is short: ask Claude for a solution, review what it built, run `tb push <slug>` when Claude tells you it's ready.

<Steps>
1. **You ask.** Describe the solution you want. Claude designs the full IoT solution for you — devices and their hierarchy, dashboards, alarms, calculated metrics, and customers and user accounts. It also generates demo telemetry emulators so your dashboards have data from day one, and validates the design against ThingsBoard as it goes.

2. **You review.** By default, Claude pauses for your approval after each design step — first the entity model, then the dashboards. You can also tell Claude to run autonomously and just show you the final result.

3. **You deploy.** When Claude says the solution is ready, run the `tb push <slug>` command it suggests. Add `--run-tasks` to also generate demo telemetry and backfill calculated fields with historical inputs:

   ```bash
   tb push smart-house --run-tasks
   ```

4. **You iterate.** Ask Claude for changes — *"add a humidity sensor to each room"*, *"split the kitchen dashboard"*. Claude regenerates the affected pieces, and you re-run `tb push`. Push is idempotent: ThingsBoard tracks entity identity by external ID, so the same solution can be pushed many times with no duplication.
</Steps>

### How to phrase your request

Claude does its best work when you mention these things up front. It will ask if you don't, but answering up front means fewer follow-up questions:

- **Domain & entities** — what's being monitored or controlled. Sensors, vehicles, machines, meters, infrastructure.
- **Hierarchy** — are these standalone devices, or are they grouped inside container assets (a building, a fleet, a site)?
- **Customers & users** — single tenant, or multiple independent customers / sites that should see only their own data?
- **Alarms & metrics** — specific thresholds or derived values you care about. Optional; Claude proposes sensible defaults if you don't specify.

**Example prompts:**

- *"Create a smart house monitoring solution with a few temperature, humidity, and motion sensors per room."*
- *"Build a fleet tracking system for two delivery customers, with location and battery alarms."*

<Aside type="tip">
The same flow is available as a slash command: `/create-solution`. Use it if you prefer that style of invocation.
</Aside>

### What Claude has access to

`/create-solution` is the umbrella command; behind the scenes Claude uses purpose-built skills for entity generation, dashboards, alarms, calculated fields, and emulators. These ship with the CLI and update automatically with it.

<Aside>
`.claude/` is regenerated by the CLI on each launch, so don't commit it — a teammate's first `tb` command after cloning will repopulate it.
</Aside>

<Aside type="caution">
Entities edited directly in the ThingsBoard UI will be **overwritten on the next `tb push`**. Treat the local project as the source of truth — or run `tb pull` first to bring UI changes back into your files.
</Aside>

## Local vs remote mode

The CLI has two operating modes:

- **Local mode** — the default after `tb init`. Commands like `tb device save` write the resource to your project files on disk; nothing reaches ThingsBoard until you run `tb push`. This is what enables managing your tenant as code, multi-target deployments, and version control of your solutions.
- **Remote mode** — `tb device save` and friends call the ThingsBoard API directly and apply changes immediately. No project files are written.

Outside any project, the CLI always talks to the API directly — there's no project on disk to write to.

Mode is a per-project setting. Switch it at any time:

```bash
tb config mode remote    # turn the current project into remote mode
tb config mode local     # switch back to the default local mode
```

To force a single command to call the API directly without changing the project's mode, add `--remote`:

```bash
tb device save --remote --json '{"name": "Sensor-01", "type": "Temperature Sensor"}'
```

## Solutions & multi-target deployment

ThingsBoard CLI treats your tenant configuration as code: entity definitions live in your project on disk, and `tb push` deploys them to ThingsBoard. The same solution can be pushed to multiple ThingsBoard targets — dev, staging, prod — without code changes.

Each entity directory under the project root holds entities that aren't part of any solution. Solutions are self-contained slices under `solutions/<slug>/` with the same directory layout in miniature.

When a solution is on disk you can target it from anywhere in the project with the `--solution` flag:

```bash
tb device save --solution smart-house --json '{"name": "Sensor-01", "type": "Temperature Sensor"}'
```

<Aside type="caution">
**Don't `cd` into solution directories** to scope commands. Always use `--solution <name>`. The `--solution` flag works from anywhere inside your project and is more reliable than relying on the current directory.
</Aside>

To start a new solution by hand (without Claude):

```bash
tb solution new smart-house
```

### Multi-target push: dev → staging → prod

The same solution can be deployed to multiple ThingsBoard environments by pushing it under different config profiles:

```bash
tb push smart-house --profile dev
tb push smart-house --profile staging
tb push smart-house --profile prod
```

The CLI writes **no state files** to your project — ThingsBoard itself tracks entity identity by external ID, so the same solution pushed to the same target twice updates the existing entities rather than creating duplicates. Develop and test against dev, promote to staging once you're happy, and ship to prod with the exact same files.

### Validating without pushing

```bash
tb validate              # validate root entities
tb validate smart-house  # validate one solution
```

Validation is a dry-run against the target ThingsBoard instance. Combine it with `--profile <name>` to validate against any environment.

### What `--run-tasks` does

`tb push --run-tasks` does the regular push and then runs two extra steps:

- **Telemetry emulators.** The CLI writes synthetic telemetry for the devices Claude set up emulators for, so dashboards have something to display.
- **Calculated field backfill.** The CLI replays historical inputs through each calculated field expression so derived values are populated for the time range covered by the emulators.

Both steps are safe to omit on subsequent pushes once the data exists — they are most useful on the very first push to a fresh tenant.

<Aside>
Claude generates emulators automatically for devices it creates. If you're driving the CLI manually, add emulators with `tb emulator add` — see `tb emulator --help` for the available subcommands.
</Aside>

### Pulling changes back from ThingsBoard

`tb pull` is the inverse of push: it fetches the current server state for entities you already track locally and rewrites the JSON files on disk. Reach for it when someone tweaked a dashboard in the UI, when you want a fresh local copy of what's running on staging, or when you need to bring a server-created entity into the project.

```bash
tb pull                                # refresh every locally-tracked entity in the current solution (auto-detected from cwd)
tb pull --solution smart-house         # same, with the solution named explicitly
tb pull dashboard --all                # refresh every dashboard in the current solution
tb pull device <uuid>                  # fetch a single entity by its server-internal UUID
tb pull device <uuid1> <uuid2>         # …or several at once
```

The UUID form is the way to pick up entities that exist on the server but aren't in the project yet — pass the entity's internal ID and the CLI writes a fresh file under the right directory.

A few flags worth knowing about:

- `--dry-run` prints the planned writes (created / updated / renamed) without touching disk. Safe to try first.
- `--with-credentials` includes device credentials in the pulled files. Off by default; opt in only when you want them committed alongside the device JSON.
- `--force` skips the safety check that aborts when target files have uncommitted git changes.

By default `tb pull` refuses to overwrite files with uncommitted local changes, so it never silently clobbers work in progress. Commit, stash, or pass `--force` to proceed.

## Custom backend extensions

Some features need server-side logic of their own — a custom REST endpoint that runs a multi-step workflow in one call, a scheduled job that combines data across entities to produce derived results, or a bridge to another system you already run. Extensions are how you add that kind of behavior. You describe what you want in plain language, Claude writes it, and `tb extension` builds, runs, and publishes it.

The full lifecycle is short: scaffold the extension once, ask Claude to build features into it, run it locally to try them out, and — when you're ready to deploy — push the image to a registry so any server can pull it.

### Scaffold the extension

Every extension has a **name**, set once when you scaffold it. The name is the local Docker image tag the CLI uses for builds (`<name>:latest`, `<name>:1.2.3`), and it shows up in the `tb extension push` examples you'll run later. Pick something descriptive — your project's slug, your company short-name, the product the extension belongs to.

Pass the name with `--name`:

```bash
tb extension new --name acme-telemetry-extension
```

If you leave `--name` off in an interactive terminal, the CLI explains what the name is for and prompts you for it:

```text
This name will be used as the local Docker image tag for your extension
(e.g. <name>:1.2.3). Pick something descriptive — your project's slug,
your company short-name, etc. You can change it later with
`tb extension rename <new-name>`.

Extension name:
```

In CI or any non-interactive session, the prompt isn't available — pass `--name` explicitly.

The CLI picks the right edition — self-hosted PE or ThingsBoard Cloud — from your active profile and writes the name into your `thingsboard.json`:

```json
{
  "extension": {
    "name": "acme-telemetry-extension"
  }
}
```

You get an `extension/` folder in your project. The one file worth keeping an eye on is `extension/README.md`. Claude keeps a plain-English summary there of every feature your extension has, plus a history of the ones you've asked it to remove.

### Building features with Claude

The flow mirrors solutions:

<Steps>
1. **You ask.** Describe the feature in plain language — what it does, who or what triggers it, what data it reads or writes. Claude picks the right kind of trigger for the job and writes the feature. Triggers Claude can wire up for you:

   - A dashboard widget calling the extension when a user clicks or interacts with it.
   - A job that runs on its own schedule (every night, every hour, etc.).
   - A web link that another system, or anyone you share the link with, can call to trigger the feature.

2. **You review.** Claude updates the features section of `extension/README.md` with a plain-English summary of what's there and how to use it.

3. **You run.**

   ```bash
   tb extension run
   ```

   The CLI builds the extension the first time, then keeps it running until you stop it.

4. **You iterate.** Ask Claude for changes — *"only invoice customers with active devices"*, *"send a daily email of the rollup instead"*. Claude regenerates the affected pieces and refreshes the README. Features you drop are moved into a "removed features" history so the story of how the extension grew stays visible.
</Steps>

<Aside type="tip">
The same flow is available as a slash command: `/create-extension`. Use it if you prefer that style of invocation.
</Aside>

<Aside type="caution">
Claude rewrites the `## Features` and `## Removed features (history)` blocks in `extension/README.md` on every iteration. Hand-edits to those two sections will be lost; everything else in the README is safe to edit.
</Aside>

### Build and run it locally

Docker is the default build target:

```bash
tb extension build              # builds acme-telemetry-extension:latest
tb extension run                # runs the image; builds first if it isn't there yet
tb extension run -d             # same, but detached (run in the background)
tb extension logs -f            # follow the running container's logs
```

When the build finishes, the CLI prints a green summary panel with the images it produced and a two-line hint pointing at the next step:

```text
Build succeeded
  • acme-telemetry-extension:latest

Next:
  tb extension run                          smoke test locally
  tb extension push --image-name <reg>/foo  publish to a registry
```

To produce a versioned image alongside `:latest`, pass `--image-tag`:

```bash
tb extension build --image-tag 1.2.3
```

That tags both `acme-telemetry-extension:1.2.3` **and** `acme-telemetry-extension:latest` — the version is the immutable artifact you'll publish, `:latest` keeps a moving pointer to your most recent build.

As soon as the container is up, the CLI prints local control hints alongside the integration paths for rule-chain and widget callers:

```text
[OK] Running 'acme-telemetry-extension' at http://localhost:8090

Local:
  tb extension logs -f       tail logs
  tb extension restart       reload after code edit
  tb extension stop          stop the container

Wire from a rule chain (REST API Call node):
  POST http://localhost:8090/api/extension/<your-endpoint>
  Header: X-Authorization: ApiKey <your-api-key>

Wire from a widget (when ThingsBoard runs on this host) — add to your
existing HAProxy frontend, before the TB backend ACL:

  acl is_extension path_beg /api/extension/
  use_backend thingsboard_extension if is_extension

  backend thingsboard_extension
      server extension 127.0.0.1:8090 check

Widget JS templates: examples/widgets/{on-premise,cloud}-button.js
```

`tb extension restart` picks up your latest changes faster than a stop + run. It auto-discovers the image that's currently running (from `docker compose ps`), so you don't need to remember which tag you last started:

```bash
tb extension restart
tb extension stop               # stop it
tb extension status             # current features, extension name, last build, whether it's running OK
```

<Aside>
Prefer the classic JAR workflow over Docker? Pass `--jar` to `build` or `run` to opt into the Maven `package` path and a local `java -jar` launch. Docker is the recommended mode — the JAR path is there for environments where Docker isn't available.
</Aside>

### Renaming your extension

If you picked a name you've outgrown — the project changed direction, you want to publish under a different brand, the original was a placeholder — rename the extension in place:

```bash
tb extension rename acme-new
```

The CLI validates the new name (same rules as `tb extension new`) and updates `thingsboard.json`. Subsequent builds tag the image as `acme-new:<tag>`.

The previous local images (`acme-old:latest`, `acme-old:1.2.3`, …) stay on your Docker daemon until you remove them yourself:

```bash
docker image rm acme-old:latest acme-old:1.2.3
```

### Switching ThingsBoard editions

The scaffolded project is pinned to one of two ThingsBoard client artifacts in its root `pom.xml` — `thingsboard-pe-client` (Professional) or `thingsboard-paas-client` (Cloud). Each exposes a slightly different set of API endpoints, so the right artifact has to be wired in before Claude generates code against it. The CLI picks one at scaffold time from your active profile; switch later with:

```bash
tb extension set-edition pe
```

The argument is `pe` or `paas`. Run `tb extension build` afterwards — the build regenerates the API reference Claude reads when writing features, so the next iteration targets the right edition's endpoints.

### Publishing the extension image

Once the extension runs the way you want, push the built image to a Docker registry so anyone — a server, a teammate, your hosting platform — can pull and run it without having the CLI installed.

`tb extension push` takes the local image (built by `tb extension build`) and uploads it to the registry/repository you pass via `--image-name`. The local name from your manifest is the **source**; `--image-name` is the **publish target** and must include any registry or org prefix.

Authenticate with the registry first — the CLI doesn't store credentials and won't run `docker login` for you:

```bash
docker login                              # Docker Hub
docker login registry.example.com         # private registry
```

Then push. To Docker Hub:

```bash
tb extension push --image-name myuser/acme-telemetry-extension
```

To a private registry:

```bash
tb extension push --image-name registry.example.com/team/acme-telemetry-extension
```

Before anything is uploaded the CLI shows a confirmation panel listing exactly what's about to leave your machine, asks you to confirm, and — after a successful push — walks you through everything you need to do on the host that runs ThingsBoard:

```text
About to push:
  myuser/acme-telemetry-extension:latest
to registry: docker.io

Push this image? [y/N]: y

Pushed
  • Pushed myuser/acme-telemetry-extension:latest

Deploy on the host that runs ThingsBoard:

  1. Copy this project's extension/deploy/ directory to that host, then cd into it
  2. cp .env.example .env
  3. Edit .env: set IMAGE_NAME=myuser/acme-telemetry-extension
       (Cloud TB: also set CORS_ALLOWED_ORIGINS, or widget calls get blocked)
  4. docker compose up -d
  5. For dashboard widgets, apply this snippet to your HAProxy frontend:

       acl is_extension path_beg /api/extension/
       use_backend thingsboard_extension if is_extension

       backend thingsboard_extension
           server extension 127.0.0.1:8090 check
```

The default answer at the prompt is **no** — pressing Enter cancels the push. Type `y` to go ahead.

By default, `tb extension push` publishes only the `:latest` tag. Pass `--image-tag` to publish a specific version alongside it — the CLI then pushes both the versioned tag and `:latest`:

```bash
tb extension push --image-name myuser/acme-telemetry-extension --image-tag 1.2.3
```

```text
About to push:
  myuser/acme-telemetry-extension:1.2.3
  myuser/acme-telemetry-extension:latest
to registry: docker.io

Push these images? [y/N]:
```

This matches the typical release pattern: each release publishes an immutable version tag *and* moves `:latest` forward so people who don't pin a version automatically pick up the newest one.

For CI pipelines and other non-interactive runs, skip the confirmation with `--yes` (or `-y`):

```bash
tb extension push --image-name myuser/acme-telemetry-extension --image-tag 1.2.3 --yes
```

<Aside type="caution">
If the local image isn't built yet, push errors out and asks you to run `tb extension build` first. The registry only ever sees what you've already built and verified locally.
</Aside>

### Deploying to your ThingsBoard host

The push hint above lists the five deploy steps; this section expands on the parts that need explaining. The same `extension/deploy/` directory works for self-hosted ThingsBoard PE and ThingsBoard Cloud.

<Steps>
1. **Copy `extension/deploy/` to the host that runs ThingsBoard.** It contains the compose file and the `.env.example` template. Use `scp`, a `git clone` on the host, or any other copy mechanism. Then `cd` into the copy on the host.

2. **Create `.env` from the template.**

   ```bash
   cp .env.example .env
   ```

   The template has every variable the compose file consumes, with inline comments explaining which ones matter for which deployment.

3. **Edit `.env` for your target.** Three variables are worth a closer look:

   - **`IMAGE_NAME`** — must match the image you just pushed (`myuser/acme-telemetry-extension`, or the full `registry.example.com/team/...` form for a private registry). Compose pulls this image when it starts the container.
   - **`THINGSBOARD_URL`** — defaults to `http://host.docker.internal:8080`, which is correct for self-hosted ThingsBoard running on the same Docker host. For ThingsBoard Cloud, uncomment the cloud line in the template and set it to your tenant's URL (e.g. `https://thingsboard.cloud` or your custom domain).
   - **`CORS_ALLOWED_ORIGINS`** — leave blank for on-premise deployments where widget calls reach the extension same-origin through HAProxy. For ThingsBoard Cloud, set it to the origin your dashboard widgets call from (typically your tenant URL).

   `TB_AUTH_API_KEY` and the username/password pair are only needed if your extension runs `@Scheduled` background jobs — leave them blank for endpoint-only extensions.

   <Aside type="caution">
   On ThingsBoard Cloud, `CORS_ALLOWED_ORIGINS` is **required** for widget callbacks. If you leave it blank, the container starts and looks healthy, but every browser request from a widget is blocked by CORS. The extension keeps defaulting to `host.docker.internal:8080` and silently fails to reach your cloud tenant from the browser. Set it before bringing the container up.
   </Aside>

4. **Start the container.**

   ```bash
   docker compose up -d
   ```

   The extension exposes port `8090` on the host. Override with `EXTENSION_PORT` in `.env` if that port is already taken.

5. **Apply the HAProxy snippet for widget integration.** If any feature in your extension is triggered from a dashboard widget, the browser needs same-origin access to `/api/extension/*`. Add the snippet from the push hint to your existing HAProxy frontend, **before** the TB backend ACL — that way HAProxy routes `/api/extension/*` to the extension service first, and only falls through to the ThingsBoard backend for everything else. A reference copy of the snippet also lives at `extension/deploy/haproxy-extension.cfg.snippet` in your project.
</Steps>

Rule-chain-triggered features don't need HAProxy at all — ThingsBoard's REST API Call node talks to the extension service directly on the same host. The HAProxy step is only there for widget callbacks, which run in the user's browser.

## Manual usage

The CLI's own help is always up to date and is the canonical reference for every command.

```bash
tb --help                # top-level command groups
tb device --help         # subcommands inside a group
tb device save --help    # flags and options for one command
```

### Verbose help with `-H`

For any command that maps to a ThingsBoard API method, append `-H` (or `--help-verbose`) to see the full request schema, including which fields are required, which are read-only, and the underlying API method name:

```bash
tb device save -H
```

Claude Code uses verbose help internally to discover schemas. Humans benefit from it too — it answers most "what JSON do I need to pass?" questions on the spot.

### A handful of representative commands

```bash
# List devices
tb device list

# Create a device by passing a JSON body
tb device save --json '{"name": "Sensor-01", "type": "Temperature Sensor"}'

# Or use the shorthand flags for common fields
tb device save --name Sensor-02 --type "Temperature Sensor"

# Save server-scope attributes on an entity
tb attributes save DEVICE 16216cd0-4f85-11f1-b03b-97f809988331 --scope SERVER_SCOPE --json '{"firmwareVersion": "1.0.3"}'
```

Every entity type has its own group (`tb dashboard`, `tb asset`, `tb customer`, `tb rule-chain`, `tb calculated-field`, ...) with the same `list` / `save` / `delete` shape.

### Shell completion

Install completion once and restart your terminal:

```bash
tb --install-completion bash    # bash
tb --install-completion zsh     # zsh
tb --install-completion fish    # fish
```

## Troubleshooting

<Aside type="note" title="Not a ThingsBoard project. Run 'tb init' first.">
You're outside any project directory. Either run `tb init` in the directory you want to manage as a project, or `cd` somewhere inside an existing project.
</Aside>

<Aside type="note" title="No entities to push.">
The targeted solution or project root contains no entity files yet. Generate some with Claude or create them manually with `tb <entity-type> save`, then push again.
</Aside>

<Aside type="tip" title="Need to see the raw HTTP exchange">
Add `--debug` to any command (for example, `tb push smart-house --debug`) or set `TB_DEBUG=1` in your environment. Both print every HTTP request and response — useful when filing a bug report.
</Aside>

<Aside type="caution" title="Authentication issues (403 on push, login failures)">
The most common cause is an API key whose owning user lacks CRUD permissions for the resources you're pushing — API keys mirror the permissions of the user they were generated for, so check that user's role and permissions in ThingsBoard. You can also re-run `tb config` to redo the wizard, or `tb config show` to confirm which profile is active.
</Aside>

## Limitations

<Aside type="caution">
- Works with **ThingsBoard PE and Cloud.**
- `tb push` has **no automatic rollback.** If you push a broken solution, fix it locally and push again.
- Entity identity is tracked server-side by **external ID**. UI edits to project-managed entities are overwritten on the next `tb push` unless you `tb pull` them back first. Brand-new entities created outside the CLI can be brought into the project with `tb pull <type> <uuid>`.
</Aside>
