ThingsBoard models the physical world as a graph of **entities** connected by **relations**. Every entity supports attributes, time-series data, and alarm rules — giving you a unified way to represent, monitor, and control your IoT infrastructure.

```plaintext
  ┌─────────────────────────────────────────────────────┐
  │                      Tenant                         │
  └──────────┬──────────────────────┬───────────────────┘
             │ Manages              │ Manages
     ┌───────▼────────┐    ┌───────▼────────┐
     │   Customer A   │    │   Customer B   │
     └───────┬────────┘    └───────┬────────┘
             │ Owns                │ Owns
     ┌───────▼────────┐    ┌───────▼────────┐
     │  Region East   │    │  Region West   │
     │    (asset)     │    │    (asset)     │
     └───────┬────────┘    └───────┬────────┘
             │ Contains            │ Contains
     ┌───────▼────────┐    ┌───────▼────────┐
     │  Field Alpha   │    │  Field Beta    │
     │    (asset)     │    │    (asset)     │
     └───┬────────┬───┘    └───┬────────┬───┘
         │        │            │        │
         ▼        ▼            ▼        ▼
     ┌───────┐ ┌───────┐  ┌───────┐ ┌───────┐
     │Sensor1│ │Sensor2│  │Sensor3│ │Sensor4│
     │(device)│ │(device)│  │(device)│ │(device)│
     └───────┘ └───────┘  └───────┘ └───────┘
```

## Core entities

| Entity | Purpose |
|--------|---------|
| **Tenant** | Top-level business entity that owns all other entities |
| **Customer** | Organization or individual that uses tenant <DocLink product={props.product} path='user-guide/devices' bold={false}>devices</DocLink> and <DocLink product={props.product} path='user-guide/assets' bold={false}>assets</DocLink> |
| **User** | Person who logs in to browse dashboards and manage entities |
| **<DocLink product={props.product} path='user-guide/devices' bold={false}>Device</DocLink>** | Physical or virtual thing that publishes telemetry and handles RPC commands |
| **<DocLink product={props.product} path='user-guide/assets' bold={false}>Asset</DocLink>** | Abstract object (building, field, vehicle) used to group and aggregate data |
| **<DocLink product={props.product} path='user-guide/dashboards' bold={false}>Dashboard</DocLink>** | Visualization of IoT data with interactive widgets |

Some entity types support **profiles** that define shared configuration for a group of similar entities:

| Profile | Applies to | Configures |
|---------|-----------|------------|
| **<DocLink product={props.product} path='user-guide/device-profiles' bold={false}>Device Profile</DocLink>** | Devices | Transport protocol, rule chain, queue, alarm rules, provisioning |
| **<DocLink product={props.product} path='user-guide/asset-profiles' bold={false}>Asset Profile</DocLink>** | Assets | Rule chain, queue, alarm rules |

## Entity data

Every entity supports three types of data:

| Data type | Description | Example |
|-----------|-------------|---------|
| **<DocLink product={props.product} path='user-guide/digital-twins/attributes' bold={false}>Attributes</DocLink>** | Static or semi-static key-value pairs | Serial number, firmware version, location |
| **<DocLink product={props.product} path='user-guide/digital-twins/time-series-data' bold={false}>Time-series</DocLink>** | Timestamped data points for storage, querying, and visualization | Temperature, humidity, battery level |
| **<DocLink product={props.product} path='user-guide/digital-twins/relations' bold={false}>Relations</DocLink>** | Directed connections between entities | Contains, Manages, Produces |

Relations help model real-world hierarchies — for example, a region that contains fields, each containing multiple sensors. See <DocLink product={props.product} path='user-guide/digital-twins/relations'>Relations</DocLink> for details on creating, querying, and using relations in rule chains and dashboards.
