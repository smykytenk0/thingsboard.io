The <a href="https://www.chirpstack.io/" target="_blank">ChirpStack</a> integration connects ThingsBoard to a ChirpStack LoRaWAN Network Server: it receives uplink messages from LoRaWAN devices, decodes them via an uplink converter, and pushes telemetry and attributes to the platform. In the reverse direction, it encodes Rule Engine messages via a downlink converter and delivers them to devices through the ChirpStack API.

## Prerequisites

You need a running ChirpStack Network Server. This guide uses a local installation via Docker Compose — see the <a href="https://www.chirpstack.io/project/guides/docker-compose/" target="_blank">ChirpStack Docker Compose guide</a>. You must also have at least one device registered and connected — see the <a href="https://www.chirpstack.io/project/guides/connect-device/" target="_blank">ChirpStack device connection guide</a>.

## Uplink converter

The uplink converter decodes the incoming ChirpStack message and maps it to the ThingsBoard data model. For the full decoder function reference, see <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter" target="_blank">Uplink data converter</DocLink>.

When adding or editing the integration, the uplink converter step offers two modes:

- **Create new** — write or paste a custom decoder script. Covered in <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter/#typed-uplink-converter" target="_blank">Typed uplink converter</DocLink>.
- **Library** — pick a vendor and model from a built-in catalog; the converter is fully pre-configured. Covered in <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter/#converters-library" target="_blank">Library</DocLink>.

<Aside type="note">From ThingsBoard 4.0, ChirpStack supports typed converters — map message fields to attributes or telemetry in the UI without writing a decoder script. Converters created before ThingsBoard 4.0 continue to work without changes.</Aside>

<Aside type="tip">
Use <DocLink product={props.product} path="user-guide/tbel">TBEL</DocLink> for converter scripts — it executes significantly faster than JavaScript inside ThingsBoard.
</Aside>

### Connection settings

<h4>Base URL</h4>

The base address of the ThingsBoard server, used to construct the HTTP endpoint URL.

<h4>HTTP endpoint URL</h4>

Auto-generated endpoint for this integration. Configure ChirpStack to forward uplink messages to this URL.

<h4>Application server URL</h4>

The address of the ChirpStack REST API service. With a standard Docker Compose installation, this is typically port `8090`.

<h4>Application server API Token</h4>

API token used to authenticate ThingsBoard requests to the ChirpStack API when sending downlink messages. See [Create a ChirpStack API key](#create-a-chirpstack-api-key).

<h4>Use API for ChirpStack 4+</h4>

Enables the ChirpStack v4 API. Enabled by default — disable only when connecting to ChirpStack 3.x.

<h4>Execute remotely</h4>

When enabled, ThingsBoard generates an **Integration key** and **Integration secret** that allow the integration to run as a separate process outside the ThingsBoard cluster — useful when the integration must reach services not accessible from the ThingsBoard server.

## Downlink converter

The downlink converter (encoder) transforms a Rule Engine message into the payload delivered to the device through the ChirpStack API. For the full encoder function reference, see <DocLink product={props.product} path="user-guide/integrations/downlink-data-converter">Downlink data converter</DocLink>.

The encoder function receives `msg`, `metadata`, and `msgType`, and must return an object with:
- `contentType` — `JSON`, `TEXT`, or `BINARY` (`BINARY` expects a Base64-encoded string)
- `data` — the encoded payload string
- `metadata` — key-value pairs forwarded to the integration; `DevEUI` and `fPort` are required by the ChirpStack API

<Aside type="note">
A downlink converter is required only when the integration sends commands to devices (RPCs, attribute updates). For uplink-only use cases it can be skipped.
</Aside>

### Add downlink converter

<Steps>
1. Go to **Integration center &#8702; Integrations** and open the ChirpStack integration.
2. Click **Toggle edit mode**.
3. In the **Downlink data converter** field, click **Create new**.
4. In the **Add data converter** dialog, enter a name, and write or paste the encoder script.
5. Click **Add**, then click **Apply changes**.
</Steps>

The rule chain contains two nodes connected in sequence:

- **get required fields** — reads the `fPort` and `devEui` client attributes from the device and copies them into message metadata. If either attribute is absent on the device, the message takes the **Failure** path and no downlink is sent. Make sure both attributes are set on every device before testing.
- **Send downlink** — forwards the message to the ChirpStack integration, which runs the downlink converter and queues the encoded payload through the ChirpStack API.

## Troubleshooting

<h3>Integration not receiving uplink messages</h3>

**Verify the HTTP endpoint URL is correct.**
Open the ChirpStack integration in ThingsBoard (**Integration center &#8702; Integrations**), click **Toggle edit mode**, and copy the **HTTP endpoint URL** from the Connection step. Then open the ChirpStack UI, go to your application's **Integrations** tab, and confirm the HTTP integration points to that exact URL.

**Verify the ChirpStack HTTP integration is active.**
In the ChirpStack UI, the HTTP integration must be listed under your application's **Integrations** tab. If it is missing, add it following the steps in [Configure integration on ChirpStack](#configure-integration-on-chirpstack).

**Check the integration Events tab.**
Go to **Integration center &#8702; Integrations**, open the ChirpStack integration, and click the **Events** tab. If uplinks arrive but show errors, the raw message body is visible there — use it to diagnose converter failures.

<h3>Device not created automatically</h3>

**Check "Allow create devices or assets" is enabled.**
Open the ChirpStack integration, click **Toggle edit mode**, and confirm **Allow create devices or assets** is turned on in the Basic settings step. When this toggle is off, ThingsBoard processes the uplink but does not create a new device if one does not already exist.

**Check the uplink converter output.**
Go to **Integration center &#8702; Data converters**, open the uplink converter, and click the **Events** tab. The **Out** column must contain a valid `deviceName` field. If the converter returns an empty object or throws an error, no device is created.

<h3>Downlink not delivered to the device</h3>

**Check the API token.**
An invalid or expired API token causes ThingsBoard to fail when calling the ChirpStack API. Open the integration, click **Toggle edit mode**, and re-enter the **Application server API Token** obtained in [Create a ChirpStack API key](#create-a-chirpstack-api-key).

**Check `DevEUI` and `fPort` in the converter output.**
Open the downlink converter **Events** tab and inspect the **Out** column. The returned `metadata` object must contain both `DevEUI` and `fPort` — the ChirpStack API requires them to queue a downlink. If either is missing, update the encoder script or add the missing client attributes (`devEui`, `fPort`) to the device.

**Check the rule chain wiring.**
Open **Rule chains &#8702; Root Rule Chain** and confirm the **check relation presence** node is connected to the **message type switch** node via the **Attributes Updated** link, and its **True** output connects to the **Downlink to Chirpstack** rule chain node. A missing or wrong link silently drops the message before it reaches the downlink converter.
