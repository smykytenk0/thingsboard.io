TCP Integration allows you to stream data from devices using a TCP transport protocol to ThingsBoard and convert device payloads to the ThingsBoard format.

<Aside type="note">
  TCP Integration can only run as a remote integration. It can run on the same machine as your ThingsBoard instance, or on a separate machine with network access to ThingsBoard.
</Aside>

<DocImage src="/src/assets/images/user-guide/integrations/tcp/tcp-integration.svg" alt="TCP integration overview diagram" />

## Prerequisites

This tutorial uses:

- A ThingsBoard Professional Edition instance running locally.
- TCP Integration running as a remote integration connected to the ThingsBoard instance.
- The `echo` command to produce a line of text, piped to `netcat` (`nc`) to send it over TCP.

The example sensor device **SN-002** publishes temperature and humidity readings to TCP Integration on port **10560**.

Three payload formats are demonstrated. Select the one that matches your device:

<Tabs>
<TabItem label="Text">
```
SN-002,default,temperature,25.7\nSN-002,default,humidity,69
```
</TabItem>
<TabItem label="JSON">
```json
[
  {
    "deviceName": "SN-002",
    "deviceType": "default",
    "temperature": 25.7,
    "humidity": 69
  }
]
```
</TabItem>
<TabItem label="Binary">
The binary payload is 25 bytes:
```
\x30\x30\x30\x30\x11\x53\x4e\x2d\x30\x30\x32\x64\x65\x66\x61\x75\x6c\x74\x32\x35\x2e\x37\x00\x00\x00
```

Byte layout:

- **Bytes 0–3** (`\x30\x30\x30\x30`) — dummy prefix bytes, skipped.
- **Byte 4** (`\x11` = decimal 17) — payload length: 17 bytes starting at byte 5.
- **Bytes 5–10** (`\x53\x4e\x2d\x30\x30\x32`) — device name: `SN-002`.
- **Bytes 11–17** (`\x64\x65\x66\x61\x75\x6c\x74`) — device type: `default`.
- **Bytes 18–21** (`\x32\x35\x2e\x37`) — temperature: `25.7`.
- **Bytes 22–24** (`\x00\x00\x00`) — dummy trailing bytes, ignored.
</TabItem>
</Tabs>

<Aside type="note">
  On the machine running TCP Integration, port **10560** must be open for incoming connections so that `nc` can connect to the TCP socket.
</Aside>

### Step 4. Connection settings

TCP Integration always runs as a remote integration — the **Execute remotely** option is enabled and cannot be changed.

By default, TCP Integration listens on port **10560**. You can change this to any available port.

Copy the **Integration key** and **Integration secret** — you will need them when configuring the remote TCP Integration service.

Select your payload type to configure the frame handler:

<Tabs>
<TabItem label="Text">

<ImageGallery images={[
  { src: '/src/assets/images/user-guide/integrations/tcp/tcp-handler-configuration-text.png', alt: 'Handler configuration — text payload type selected' },
  { src: '/src/assets/images/user-guide/integrations/tcp/tcp-integration-setup-text-4.png', alt: 'Connection settings — text payload handler configuration' },
]} />

Set the following handler values:

- **Max Frame Length** — maximum length of a decoded frame; leave the default **128**.
- **Strip Delimiter** — enable this to drop the newline delimiter from the payload.
- **Message Separator** — set to **System Line Separator** (newline is used as the delimiter).

</TabItem>
<TabItem label="JSON">

<ImageGallery images={[
  { src: '/src/assets/images/user-guide/integrations/tcp/tcp-handler-configuration-json.png', alt: 'Handler configuration — JSON payload type selected' },
  { src: '/src/assets/images/user-guide/integrations/tcp/tcp-integration-setup-json-4.png', alt: 'Connection settings — JSON payload handler configuration' },
]} />

</TabItem>
<TabItem label="Binary">

<ImageGallery images={[
  { src: '/src/assets/images/user-guide/integrations/tcp/tcp-handler-configuration-binary.png', alt: 'Handler configuration — binary payload type selected' },
  { src: '/src/assets/images/user-guide/integrations/tcp/tcp-integration-setup-binary-4.png', alt: 'Connection settings — binary payload handler configuration' },
]} />

Set the following handler values to match the sample binary payload:

- **Max Frame Length** — leave the default **128**.
- **Length Field Offset** — **4** (the length field is byte 4: `\x30\x30\x30\x30`**`\x11`**`\x53…`).
- **Length Field Length** — **1** (the length field is 1 byte).
- **Length Adjustment** — **0** (no compensation needed; the length field already contains the correct value of 17).
- **Number of first bytes to strip** — **5** (skip the 4 dummy bytes + the 1 length byte to reach the data: `\x30\x30\x30\x30\x11`**`\x53\x4e\x2d…`**).

</TabItem>
</Tabs>

Additional socket options (left at defaults for this tutorial):

- **Max number of pending connects** — maximum backlog queue length for incoming connections.
- **Size of the buffer for inbound socket** — receive buffer size in KB.
- **Size of the buffer for outbound socket** — send buffer size in KB.
- **Enable keep-alive messages** — sends periodic probes to keep the connection alive.
- **Disable Nagle's buffering algorithm** — sends data immediately without buffering.

Click **Add** to save the integration.

To send messages from ThingsBoard back to a device, add a downlink converter to the integration and configure a rule chain.

  We recommend using <DocLink product={props.product} path="user-guide/tbel">TBEL</DocLink> for converter scripts — it executes much more efficiently than JavaScript inside ThingsBoard.
