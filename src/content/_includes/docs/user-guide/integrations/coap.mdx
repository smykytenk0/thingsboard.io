CoAP Integration connects ThingsBoard to devices that use the [CoAP protocol](https://tools.ietf.org/html/rfc7252). It exposes a CoAP endpoint, receives device payloads, and transforms them into ThingsBoard telemetry and attributes via an uplink converter.

<DocImage src="/src/assets/images/user-guide/integrations/coap-integration.svg" alt="CoAP Integration architecture diagram — device sends CoAP messages to the ThingsBoard CoAP endpoint, the uplink converter decodes the payload and pushes telemetry and attributes to the platform" />

<Aside type="caution">
The CoAP endpoint is served by the **tb-core** service and must be reachable from the device network. For DTLS modes, additional key store configuration is required — see [Connection settings](#connection-settings).
</Aside>

## Prerequisites

In this tutorial, device **SN-001** sends temperature and humidity readings to the CoAP Integration at `coap://localhost`. 

The integration accepts three payload types:

<Tabs>
<TabItem label="Text">
```
SN-001,default,temperature,25.7,humidity,69
```
</TabItem>
<TabItem label="JSON">
```json
{
  "deviceName": "SN-001",
  "deviceType": "default",
  "temperature": 25.7,
  "humidity": 69
}
```
</TabItem>
<TabItem label="Binary">
Hex string:
```
\x53\x4e\x2d\x30\x30\x31\x64\x65\x66\x61\x75\x6c\x74\x32\x35\x2e\x37\x36\x39
```

Byte layout:

| Bytes | Hex | Value |
|-------|-----|-------|
| 0–5 | `\x53\x4e\x2d\x30\x30\x31` | Device name: `SN-001` |
| 6–12 | `\x64\x65\x66\x61\x75\x6c\x74` | Device type: `default` |
| 13–16 | `\x32\x35\x2e\x37` | Temperature: `25.7` |
| 17–18 | `\x36\x39` | Humidity: `69` |
</TabItem>
</Tabs>

## ThingsBoard setup

### Step 1. Create uplink converter

The uplink converter decodes the CoAP message body and maps it to the ThingsBoard data model. 

The decoder function receives:
- `payload` — raw bytes of the CoAP message body
- `metadata` — `integrationName` and any key-value pairs configured in the integration's Metadata settings

Unlike HTTP or MQTT, CoAP provides no transport-level metadata (no headers, no topic) — `deviceName` and `deviceType` must be derived from the payload itself.

It must return `deviceName` and `deviceType` (required); `telemetry` and `attributes` are optional flat key-value maps — nested objects are not supported.

For the full decoder function reference, see <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter">Uplink data converter</DocLink>.

#### Connection settings

<h4>Security mode</h4>

Defines the transport security for the CoAP connection:

| Mode | Description |
|------|-------------|
| **NO SECURE** (default) | Plain CoAP over UDP — no encryption |
| **DTLS** | CoAP over DTLS 1.2 — encrypted |
| **MIXED** | Both plain and DTLS endpoints active simultaneously |

Both **CON** (Confirmable) and **NON** (Non-Confirmable) CoAP message types are supported in all modes. ThingsBoard acknowledges CON messages automatically.

For **DTLS** and **MIXED** modes, enable DTLS support in `thingsboard.yml` or set the following environment variables:

```bash
# Enable/disable DTLS 1.2 support
export COAP_DTLS_ENABLED=true
# Default CoAP DTLS bind port
export COAP_DTLS_BIND_PORT=5484
# Path to the key store that holds the SSL certificate
export COAP_DTLS_KEY_STORE=coapserver.jks
# Password used to access the key store
export COAP_DTLS_KEY_STORE_PASSWORD=server_ks_password
# Password used to access the key
export COAP_DTLS_KEY_PASSWORD=server_key_password
# Key alias
export COAP_DTLS_KEY_ALIAS=serveralias
# Skip certificate validity check for client certificates
export TB_COAP_X509_DTLS_SKIP_VALIDITY_CHECK_FOR_CLIENT_CERT=false
```

<Aside type="note">
The values above are defaults. At minimum, update the key store path and credentials before enabling DTLS. See the <a href="/docs/reference/coap-api/getting-connected/#coap-over-dtls--server-configuration">CoAP over DTLS</a> guide for full configuration details.
</Aside>

<h4>Base URL</h4>

The base address of the ThingsBoard CoAP endpoint.

Example: `coap://int.my.thingsboard`

<h4>Path</h4>

Optional URL path appended to the base URL.

Example: `coap://int.my.thingsboard/my-path`

<h4>CoAP endpoint URL</h4>

Auto-generated from the **Base URL** and the path `/i/$INTEGRATION_ROUTING_KEY`. The `/i/` prefix is a fixed segment that routes incoming CoAP messages to the integration; the routing key (UUID) is assigned at creation time. Devices must POST to this URL.

<h4>Execute remotely</h4>

When enabled, ThingsBoard generates an **Integration key** and **Integration secret**. These credentials allow the integration to run as a separate process — either on a remote server or at the edge — and communicate with ThingsBoard over a secure channel. Use this option for deployments where the integration service must reside outside the main ThingsBoard cluster (e.g., in a DMZ or on-premises gateway).

## Send Uplink Message

Once the integration is created, the CoAP server registers the endpoint and waits for incoming data. Send a test message using `coap-client` (part of [`libcoap`](https://libcoap.net/install.html)), replacing `$YOUR_COAP_ENDPOINT_URL` with the URL copied during setup:

<Tabs>
<TabItem label="Text payload">

```bash
echo -e 'SN-001,default,temperature,25.7,humidity,69' | coap-client -m post $YOUR_COAP_ENDPOINT_URL -t text/plain -f-
```
