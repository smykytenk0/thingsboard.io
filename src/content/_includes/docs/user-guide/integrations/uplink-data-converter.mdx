The **Uplink data converter** is the decoding stage of every **ThingsBoard integration**, where raw payloads from an external device or network server are mapped to the ThingsBoard data model: device or asset name, profile, customer, group, telemetry values, and attributes.

ThingsBoard provides three ways to define the decoder:
- **Typed:** The decoder is auto-generated and is specific to each integration type, taking into account the data format and semantics of that integration.   
  Supported integrations: <DocLink product={props.product} path="user-guide/integrations/chirpstack" target="_blank">ChirpStack</DocLink>, <DocLink product={props.product} path="user-guide/integrations/loriot/" target="_blank">Loriot</DocLink>, <DocLink product={props.product} path="user-guide/integrations/ttn/" target="_blank">The Things Stack Community</DocLink>, <DocLink product={props.product} path="user-guide/integrations/tti/" target="_blank">The Things Stack Industries</DocLink>, <DocLink product={props.product} path="user-guide/integrations/thingpark/" target="_blank">ThingPark</DocLink>, and <DocLink product={props.product} path="user-guide/integrations/thingpark-enterprise" target="_blank">ThingPark Enterprise</DocLink>.
- **Generic:** Custom script, works with any integration type.
- **Library:** Choose a vendor and model from a catalog pre-built decoders for 100+ devices.

Each integration uses one converter. A converter can be shared across multiple integrations — changes apply immediately to all.

## Creating a converter

The uplink converter is configured in **Step 2** of the <DocLink product={props.product} path="user-guide/integrations/#creating-an-integration" target="_blank">integration wizard</DocLink>:

- **Previously created converter** — select an existing converter.
- **Create new** — define the decoder inline within the wizard.
- **Library** — choose a vendor and model from the built-in catalog (supported integration types only).

The converter is saved to **Integration center &#8702; Data converters** when the integration is saved. You can also create converters there in advance.

## Generic uplink converter

A generic converter works with any integration type and gives you full control over the decoding logic through a custom script.

<Aside type="tip">
Use <DocLink product={props.product} path="user-guide/tbel">TBEL</DocLink> for converter scripts — it executes significantly faster than JavaScript inside ThingsBoard.
</Aside>

<Steps>
1. Go to **Integration center &#8702; Data converters**.
2. Click **+ Add data converter &#8702; Create new converter**.
3. Set **Converter type** to **Uplink** (default).
4. Select an **Integration type**, or leave it as **All** to apply the converter across all integration types.
5. Enter a **Name**.
6. Write or paste the decoder script in **Main decoding configuration**.
7. Click <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter/#test-payload-decoder">**Test decoder function**</DocLink> to validate.
8. Optionally add keys to the **Update only keys list** in **Advanced decoding parameters**.
9. Click **Add**.
</Steps>

### Decoder function

The decoder function receives two parameters:

| Parameter | Description |
|-----------|-------------|
| `payload` | Raw message payload. Type depends on content type: **JSON** object, UTF-8 string (**Text**), or byte array (**Binary (Base64)**). |
| `metadata` | Key-value map with integration-level fields. `integrationName` is always present. Additional keys can be defined in the integration's metadata settings. |

<Aside type="note">
  For the **Binary (Base64) payload content type**, the payload is automatically decoded into a byte array, so there's no need to decode it manually in your script.
</Aside>

For binary payloads, TBEL provides built-in helpers: `parseBytesToInt`, `parseBytesToFloat`, `bytesToHex`, `decodeToString`, and others. See the <DocLink product={props.product} path="user-guide/tbel">TBEL reference</DocLink> for the full list.

The function must return an object with these fields:

| Field | Required | Description |
|-------|----------|-------------|
| `deviceName` | Yes (device) | Name of the device to create or update. |
| `deviceType` | Yes (device) | Device profile name. |
| `assetName` | Yes (asset) | Name of the asset (alternative to `deviceName`). |
| `assetType` | Yes (asset) | Asset profile name. |
| `customerName` | No | The entity is assigned to this customer. ThingsBoard creates the customer if it does not exist. |
| `groupName` | No | The entity is added to this entity group. ThingsBoard creates the group if it does not exist. |
| `attributes` | No | Key-value map of attribute values. Values must be scalars — nested objects are not supported. |
| `telemetry` | No | Key-value map of telemetry values, or a `{ ts, values }` object with an explicit timestamp, or an array of `{ ts, values }` objects to report multiple timestamps in one message. Values must be scalars — nested objects are not supported. |

Example decoder:

```js
var data = decodeToJson(payload);
return {
    deviceName: data.deviceId,
    deviceType: 'thermometer',
    attributes: { serialNumber: data.sn },
    telemetry: {
        ts: data.timestamp,
        values: { temperature: data.temp, humidity: data.rh }
    }
};
```

Multiple telemetry timestamps example:

```js
telemetry: [
    { ts: data.ts1, values: { temperature: data.temp1 } },
    { ts: data.ts2, values: { temperature: data.temp2 } }
]
```

### Update only keys list

Keys added to the **Update only keys list** in **Advanced decoding parameters** are written to the database only when their value differs from the previous message.

<Aside type="note">
  In a cluster deployment, keys in the <strong>Update only keys list</strong> may still be written more than once when uplink messages arrive at different executor nodes, or immediately after a converter configuration change.
</Aside>

## Typed uplink converter

Available from **ThingsBoard 4.0**. Configure the entity type, name pattern, and key mappings in the UI — ThingsBoard auto-generates the decoder script for the selected integration type.

Supported integrations: <DocLink product={props.product} path="user-guide/integrations/chirpstack" target="_blank">ChirpStack</DocLink>, <DocLink product={props.product} path="user-guide/integrations/loriot/" target="_blank">Loriot</DocLink>, <DocLink product={props.product} path="user-guide/integrations/ttn/" target="_blank">The Things Stack Community</DocLink>, <DocLink product={props.product} path="user-guide/integrations/tti/" target="_blank">The Things Stack Industries</DocLink>, <DocLink product={props.product} path="user-guide/integrations/thingpark/" target="_blank">ThingPark</DocLink>, and <DocLink product={props.product} path="user-guide/integrations/thingpark-enterprise" target="_blank">ThingPark Enterprise</DocLink>.

<Aside type="note">Converters created before ThingsBoard 4.0 continue to work without changes.</Aside>

<Steps>
1. Go to **Integration center &#8702; Data converters**.
2. Click **+ Add data converter &#8702; Create new converter**.
3. Set **Converter type** to **Uplink** (default) and select a supported **Integration type** from the dropdown.
4. In the **Uplink data converter** section, select the **Create new** tab and enter a **Name**.
5. In <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter/#main-decoding-configuration">**Main decoding configuration**</DocLink>, set the **Entity type** and **name** pattern (for example, `Device $eui`). Review the auto-generated decoder, then click <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter/#test-payload-decoder">**Test payload decoder**</DocLink> to validate.
6. Optionally expand <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter/#advanced-decoding-parameters">**Advanced decoding parameters**</DocLink> to adjust device profile, label, customer, group, and key mappings.
7. Click **Add**.
</Steps>

### Main decoding configuration

**Entity type:** The type of entity the converter creates or updates: **Device** or **Asset**.

**Device/Asset name:** Name of the entity. Use `$field` references to substitute values from the incoming message — for example, `Device $eui`. All supported integration types expose at least `$eui`. Refer to the integration-specific guide for the full list of available fields.

**Decoder script:** Auto-generated TBEL or JavaScript function that parses the raw payload and returns `{ attributes, telemetry }`. You can modify or fully replace it.

The function receives:
- `payload` — raw payload as a byte array.
- `metadata` — message fields: `ts` (timestamp in ms), `eui`, `devAddr`, `fPort`, `adr`, `dr`, `frequency`, `rssi`, `snr`, and others depending on the integration type.

It must return:

```js
return {
    attributes: { key: 'value' },
    telemetry: { ts: metadata.ts, values: { temperature: 23.5 } }
};
```

Do not return `deviceName` or `deviceType` — ThingsBoard resolves those from the entity type and name pattern above.

### Advanced decoding parameters

All fields are pre-populated for the selected integration type. Adjust as needed.

**Device profile name:** Profile assigned to the created or updated device. Supports `$field` substitution.

**Device label:** Optional device label. Supports `$field` substitution.

**Customer name:** If set, the device is assigned to this customer.

**Device group name:** If set, the device is added to this entity group.

**Telemetry:** Keys from the decoded output to save as telemetry. Leave empty to save all values returned by the decoder.

**Attributes:** Keys from the decoded output to save as attributes.

**Update only keys list:** Keys written to the database only when their value has changed since the previous message.

## Test payload decoder

Launches the built-in testing tool, which lets you run the decoder against a sample message without connecting a real device.  
Use it to verify that the script produces the expected attributes and telemetry before deployment.

- **Generic converter:** click **Test decoder function** below the script.
- **Typed converter:** click **Test payload decoder** below the script.

**Payload content type:** Encoding of the sample payload:
- **JSON:** parsed as a JSON object.
- **Text:** plain UTF-8 string.
- **Binary (Base64):** Base64-encoded bytes, decoded to a byte array automatically.

**Payload:** Sample input for the decoder. Pre-filled with a representative example for the selected integration type.

**Decoder script:** The `function payloadDecoder(payload, metadata)` script loaded from the converter. You can edit it directly in the test dialog to try changes without saving the converter.

**Metadata:** Key-value pairs available via the `metadata` object. Pre-populated from the last received message if available; edit entries to simulate different scenarios. `metadata.ts` holds the message timestamp in milliseconds.

**Updated RAW payload** — clicking this button in the top-right corner opens a panel with the full raw JSON of the last received message.It contains all fields ThingsBoard receives at the integration endpoint.
You can edit the payload directly here and click **Apply** to use it as the new test input — useful for simulating specific field values or edge cases without sending real device traffic.

## Import and export

Converters can be exported as JSON and imported into any ThingsBoard instance, which is useful for sharing custom decoders across environments or team members.

**Export:** On the **Data converters** list, click the export icon on a converter row. Alternatively, open the converter details and click **Export converter** in the panel header. Save the downloaded JSON file.

**Import:** Click **+ Add data converter > Import converter**, drag and drop the JSON file or click **Browse file** to select it, then click **Import**. All settings are restored: name, type, integration type, script, and advanced parameters.

## Managing data converters

All converters are listed in **Integration center &#8702; Data converters**. From the list you can:

- **Edit**: click a converter name to open it, make changes, and click **Apply changes**.
- **Delete**: click the delete icon on a row, or select multiple converters and use bulk delete.

A converter is referenced by ID. If the same converter is assigned to multiple integrations, any change to the converter (script, advanced parameters) immediately affects all integrations using it.
