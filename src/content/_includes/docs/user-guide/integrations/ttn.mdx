The Things Stack Community (TTN) is a LoRaWAN network designed for connecting devices using the LoRaWAN stack.
After integrating The Things Stack with ThingsBoard, you can connect, communicate, process, and visualize data from devices in the ThingsBoard IoT platform.

## The Things Stack Community setup

### Step 1. Register an application

<Steps>
1. Go to the <a href="https://console.thethingsnetwork.org/" target="_blank">TTN console</a>, open the **Applications** section, and press **Add application**.
2. Fill in the required fields:
   - **Application ID** — e.g. `thingsboard-connection`
   - **Handler registration** — identifies the region where the application will be registered (e.g. `eu1` for the EU region)
3. Press **Create application**.
</Steps>

### Step 2. Register a device

<Steps>
1. In the application, open the **Devices** page and press **Register device**.
2. Fill in the required fields:
   - **Device ID** — e.g. `thermostat-a`
   - **Device EUI** — press **Generate** to create a random identifier
   - **AppEUI** — can be filled with zeros
   - **AppKey** — press **Generate** to create a random identifier
3. Press **Register**.
</Steps>

### Step 3. Generate an API access key

An access key is required to authenticate the ThingsBoard integration with TTN.

<Steps>
1. In the TTN console, go to **API keys** in your application menu.
2. Press **Add API key**, enter a name, and select the required permissions.
3. Press **Generate key** and **copy the key immediately** — it will not be shown again.
</Steps>

<Steps>

    </li>
    <li>

      </ul>

    <li>
      <span><strong>Connection settings:</strong></span>
      <ul>
        <li><strong>Region</strong> — the region where your TTN application is registered (e.g. <code>eu1</code>).</li>
        <li><strong>Port</strong> — <code>8883</code>.</li>
        <li><strong>Credentials</strong>:</li>
        <ul>
          <li><strong>Username</strong> — the Application ID from TTN (e.g. <code>my-thingsboard@ttn</code>).</li>

    </li>
    <li>Optionally click <strong>Check connection</strong> — the wizard advances to a verification step showing <strong>Connected</strong> when successful.</li>
    <li>Click <strong>Add</strong> to complete the integration setup.</li>
  </ol>
</Steps>

### Connection settings

<h4>Host type</h4>

| Value | Description |
|-------|-------------|
| **Region** | Connect to a standard TTN cluster. Enter the **Region** identifier (e.g. `eu1`) — ThingsBoard constructs the host as `{region}.cloud.thethings.network`. |
| **Custom** | Specify a fully custom broker hostname (e.g. for a private TTN deployment). |

<h4>Port</h4>

| Parameter | Default | Description |
|-----------|---------|-------------|
| **Port** | `8883` | MQTT broker port. `8883` is the standard TLS port used by TTN. |

<h4>Credentials</h4>

| Parameter | Description |
|-----------|-------------|
| **Username** | The TTN Application ID (e.g. `my-thingsboard`). |
| **Password** | The API access key generated in your TTN application. |

<h4>Enable SSL</h4>

Encrypts the MQTT connection using TLS. Enabled by default and required for connections to TTN public clusters.

<h4>Use API v3</h4>

Switches the MQTT topic structure between TTN API v2 and v3.

| Value | Topic format |
|-------|--------------|
| **Enabled** | `v3/{appId}/devices/{devId}/up` (TTN v3 topic format) |
| **Disabled** | `+/devices/+/up` (TTN v2 topic format) |

<h4>Topic filters</h4>

Defines which MQTT topics ThingsBoard subscribes to for uplink messages. The default filter is pre-configured based on the **Use API v3** setting.

| Field | Description |
|-------|-------------|
| **Topic** | MQTT topic filter. `+` matches a single topic level. Automatically set based on the API version. |
| **QoS** | `0` — at most once (fire and forget); `1` — at least once (may deliver duplicates); `2` — exactly once |

<h4>Execute remotely</h4>

When enabled, ThingsBoard generates an **Integration key** and **Integration secret** that allow external services to push messages into this integration via the Integration API.

<h4>Advanced settings</h4>

| Parameter | Default | Description |
|-----------|---------|-------------|
| **Protocol version** | `MQTT 3.1.1` | MQTT protocol version used for the broker connection. |
| **Max bytes in message** | `32368` | Maximum payload size in bytes. Messages exceeding this limit are dropped. |
| **Connection timeout (sec)** | `10` | Seconds ThingsBoard waits for a broker response before marking the connection as failed. |
| **Downlink topic pattern** | `${applicationId}/devices/${devId}/down` | MQTT topic used for publishing downlink messages to TTN devices. `${applicationId}` and `${devId}` are resolved from the downlink converter output metadata. |

## Validate downlink messages

### Step 1. Add downlink data converter

The downlink converter encodes ThingsBoard commands into TTN-compatible messages. The output must contain a `downlinks` array with `f_port`, `frm_payload`, and `priority` fields.

**Expected output structure:**
```json
{
  "downlinks": [{
    "f_port": 2,
    "frm_payload": "vu8=",
    "priority": "NORMAL"
  }]
}
```

The result object must follow this format:
- `contentType` — how data is encoded: `TEXT`, `JSON`, or `BINARY`
- `data` — the actual payload sent to the device. See the <a href="https://www.thethingsnetwork.org/docs/applications/mqtt/api" target="_blank">TTN API documentation</a> for details.
- `metadata` — must include `devId` to identify the target device in TTN

This converter reads the `powerState` field from the incoming ThingsBoard message and sends it as a base64-encoded payload to device `thermostat-a`.

Add the downlink converter to the existing integration:

  <Steps>
  <ol>
    <li>Go to <strong>Integrations center &#8702; Integrations</strong> and open the <strong>The Things Stack Community integration</strong>.</li>
    <li>Click <strong>Toggle edit mode</strong>.</li>
    <li>In the <strong>Downlink data converter</strong> field, click <strong>Create new</strong>.</li>
    <li>Enter converter name.</li>
  <li>
    <p>Paste the encoder function by copying it below:</p>
    <Tabs>
      <TabItem label="TBEL (Recommended)">
        ```js maxLines=15 collapsible
        var data = {
        downlinks: [{
        f_port: 2,
        confirmed: false,
        frm_payload: btoa(msg.powerState),
        priority: "NORMAL"
      }]
      };

        var result = {
        contentType: "JSON",
        data: JSON.stringify(data),
        metadata: {
        devId: 'thermostat-a'
      }
      };

        return result;
        ```
      </TabItem>
      <TabItem label="JavaScript">
        ```js maxLines=15 collapsible
        var data = {
        downlinks: [{
        f_port: 2,
        confirmed: false,
        frm_payload: btoa(msg.powerState),
        priority: "NORMAL"
      }]
      };

        var result = {
        contentType: "JSON",
        data: JSON.stringify(data),
        metadata: {
        devId: 'thermostat-a'
      }
      };

        return result;
        ```
      </TabItem>
    </Tabs>
  </li>
    <li>Click <strong>Add</strong> to save the converter, then click <strong>Apply changes</strong> to update the integration.</li>
  </ol>
  </Steps>

### Step 2. Update Root Rule Chain

To test downlink, configure the Root Rule Chain to send a downlink message when a device attribute is changed.

<Steps>
1. Open **Rule Chains** → **Root Rule Chain** and click the edit icon.
2. Find the **Integration Downlink** node in the node panel and drag it onto the canvas.
3. Set the **Name**, select **The Things Stack Community integration**, and click **Add**.
4. Connect the **Message Type Switch** node to the **Integration Downlink** node using the **Attributes Updated** relation.
5. Click **Apply changes** to save the rule chain.
</Steps>

### Step 3. Create shared attribute

Now trigger a downlink message by adding a shared attribute:

<Steps>
1. Go to **Devices** → **All** → **thermostat-a** → **Attributes** tab.
2. Switch to **Shared attributes** and click **+** to add a new attribute.
3. Enter the key `powerState` and value `on`, then click **Add**.
</Steps>

<DocImage src="/src/assets/images/user-guide/integrations/ttn/ttn-validate-downlink-add-attribute-1.png" alt="ThingsBoard — Add shared attribute dialog: key powerState, value on, on device thermostat-a" />

This triggers the downlink message to the device `thermostat-a` containing the `powerState` field value. Open the TTN console, navigate to the application, and check the **Live data** tab — the downlink message should appear as received.
