ThingsBoard dashboards provide interactive visualization for monitoring and managing IoT data. Key capabilities include data visualization through charts, graphs, and tables; device management; interactive control elements; customizable layouts; and role-based access controls.

A ThingsBoard dashboard is built from five core building blocks. Understanding how they connect will help you design effective dashboards.

The following diagram shows a typical drill-down dashboard. The **Main** state lists all devices in an Entity Table. When a user clicks a row, an **action** fires and navigates to the **Device Details** state, where a chart shows data for the selected device. Each state has its own **layout** and **time window**, and each widget gets data through an **alias**.

```text
┌─ Dashboard ───────────────────────────────────────────────┐
│                                                           │
│  ┌─ State: Main ────────────────────────────────────┐     │
│  │                                                  │     │
│  │  Layout: Default                                 │     │
│  │  ┌──────────────────────────────────────────┐    │     │
│  │  │  Widget: Entity Table                    │    │     │
│  │  │  Data source: Alias "All Devices"        │    │     │
│  │  │  Time window: Real-time, last 1 hour     │    │     │
│  │  │                                          │    │     │
│  │  │  Row click ──► Action ───────────────────┼────┼──┐  │
│  │  └──────────────────────────────────────────┘    │  │  │
│  └──────────────────────────────────────────────────┘  │  │
│                                                        │  │
│  ┌─ State: Device Details ─────────────────────────┐   │  │
│  │                                                 │   │  │

│  │  ┌──────────────────────────────────────┐       │      │
```

## States

States are named views within a single dashboard that form a navigation hierarchy. Every dashboard has at least one state — the **root state**. You can add additional states to create detail pages, configuration panels, or any other sub-views.

**When to use:** Drill-down from a device/asset list to individual entity details without leaving the dashboard. For example, a root state lists all thermostats, and a detail state shows temperature charts for the selected thermostat.

```text
┌─────────────┐    row click    ┌──────────────────┐
│  Root state  │ ─────────────► │  Detail state     │
│  (all devs)  │                │  (selected dev)   │
│              │ ◄───────────── │                    │
└─────────────┘    back button  └──────────────────┘
```

Learn how to create and manage states in the <DocLink product={props.product} path="user-guide/dashboards">Dashboards</DocLink> guide.

## Entity aliases

An alias is a named reference that tells a widget where to get data. Aliases can be **static** (pointing to a fixed entity) or **dynamic** (resolved at runtime from dashboard state, relations, device type, etc.).

Not every widget requires an alias — static widgets like HTML cards or navigation buttons may have no data source. But any widget that displays entity data (telemetry, attributes, alarms) needs at least one.

**When to use:** Use static aliases for dashboards that always show the same entity. Use dynamic aliases — especially "Entity from dashboard state" — for drill-down scenarios where the alias resolves to whichever entity the user selected in the parent state.

```text
┌─ Alias: "All Devices" ─────────┐      ┌─ Widget ────────┐
│  Type: Device type              │      │                  │
│  Device type: thermostat        │ ───► │  Entity Table    │
│                                 │      │  (shows all      │
│  Resolves to:                   │      │   thermostats)   │
│   • Thermostat A                │      └──────────────────┘
│   • Thermostat B                │
│   • Thermostat C                │
└─────────────────────────────────┘

┌─ Alias: "Selected Device" ─────┐      ┌─ Widget ────────┐
│  Type: Entity from dashboard    │      │                  │
│        state                    │ ───► │  Line Chart      │
│                                 │      │  (shows selected │
│  Resolves to: entity passed     │      │   device data)   │
│  by the action from root state  │      └──────────────────┘
└─────────────────────────────────┘
```

## Layouts

Layouts control how widgets are arranged on the dashboard grid. Each <DocLink product={props.product} path="user-guide/dashboards">state</DocLink> has its own layout. Three types are available:

- **Default** — widgets snap to a column-based grid. Supports drag-and-drop positioning. If <DocLink product={props.product} path="user-guide/layouts">auto-fill layout height</DocLink> is enabled, widgets redistribute vertically to fill the screen.
- **<DocLink product={props.product} path="user-guide/scada" bold={false}>SCADA</DocLink>** — designed for industrial HMI screens. No margin between widgets, so pipes and SCADA symbols always connect properly. Widgets are placed with a transparent background and no shadow by default, blending into the SCADA canvas.
- **Divider** — splits the dashboard into left and right panels, useful for master-detail views (e.g., entity list on the left, entity details on the right).

```text
Default               SCADA                Divider
┌──────────────┐    ┌──────────────┐    ┌───────┬───────┐
│ ┌────┐┌────┐ │    │  ┌──┐        │    │       │       │
│ │    ││    │ │    │  └──┘──┐     │    │ Left  │ Right │
│ └────┘└────┘ │    │       │ │    │    │ panel │ panel │
│ ┌──────────┐ │    │ ┌─────┘ │    │    │       │       │
│ │          │ │    │ └───────┘    │    │       │       │
│ └──────────┘ │    │  no margins  │    │       │       │
└──────────────┘    └──────────────┘    └───────┴───────┘
column grid         seamless canvas     split panels
```

Each layout also supports <DocLink product={props.product} path="user-guide/layouts">breakpoints</DocLink> — screen-width thresholds that let you define different widget arrangements for desktop, tablet, and mobile. This means a single state can adapt its widget set and positioning for different screen sizes.

See <DocLink product={props.product} path="user-guide/layouts">Layouts</DocLink> for layout settings, breakpoints, and configuration details.

## Actions

Actions are event handlers that respond to user interactions — a row click, a header button press, a map marker tap — and trigger navigation, state updates, URL opens, or custom JavaScript logic.

**When to use:** Wire up drill-down navigation between states, link widgets to external pages, or trigger custom functions (delete entity, open dialog, etc.).

```text
Trigger              Action type               Result
─────────────────    ──────────────────────    ─────────────────
Row click         →  Navigate to new state  →  Open detail state
Header button     →  Navigate to dashboard  →  Open another dashboard
Cell button       →  Custom action          →  Run JS function
Marker click      →  Open URL               →  Open external page
```

See <DocLink product={props.product} path="user-guide/actions">Widget Actions</DocLink> for the full list of action types and action sources.

## Time windows

The time window controls the time range and aggregation for all time-series data on the dashboard. Two modes are available:

- **Real-time** — a sliding window that continuously updates (e.g., "last 1 hour"). Use for live monitoring dashboards.
- **History** — a fixed time range loaded once (e.g., "March 1 – March 7"). Use for analysis and reporting.

```text
Real-time (sliding window)          History (fixed range)

  ◄──── last 1 hour ────►  now     start              end
  │▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓│       │▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓│
  ·───────────────────────► ·       Mar 1            Mar 7
    window moves with time            fixed range
```

The dashboard-level time window applies to all widgets by default. Individual widgets can override it with their own time window settings. See the <DocLink product={props.product} path="user-guide/dashboards">Dashboards</DocLink> guide for time window configuration details.
